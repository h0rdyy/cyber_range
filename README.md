# CyberRange MVP

Лёгкий веб-стенд для управления учебными уязвимыми лабораториями в Docker.
Оператор запускает и останавливает лабы одной кнопкой, видит их статус и
доступность, а также нагрузку на систему. Написан на Go, работает как один
самодостаточный бинарник.

> ⚠️ **Предупреждение по безопасности.** Лаборатории намеренно уязвимы
> (DVWA, OWASP Juice Shop и т.п.). Их порты по умолчанию публикуются только
> на `127.0.0.1`, чтобы стенды не торчали в сеть. Не запускайте это на
> машине с публичным IP и не меняйте привязку на `0.0.0.0`, если не
> понимаете последствий. Используйте только в изолированном окружении.

---

## Возможности

- Запуск/остановка лабораторий из веб-интерфейса (dashboard и admin-панель).
- Конфигурация лабов через YAML — без пересборки приложения.
- Живые метрики системы (CPU, RAM) и статус каждого контейнера.
- Проверка доступности стенда **со стороны хоста** — по опубликованному
  порту, а не только по внутреннему Docker HEALTHCHECK (надёжнее для
  образов без `curl`/`wget`).
- Фоновый монитор кэширует метрики, поэтому `/api/status` отвечает мгновенно.
- Корректное завершение по `Ctrl+C` (graceful shutdown).
- Статистика запусков по каждому лабу.

## Требования

- Go 1.21 или новее (для сборки).
- Доступ к работающему Docker-демону (варианты установки — ниже).

Приложение **кроссплатформенное** — собирается и работает под Linux, Windows
и macOS. Нужен доступ к Docker-демону; как именно его обеспечить — зависит от
того, откуда запускается приложение:

- **Запуск в Linux или в WSL** — достаточно установить Docker Engine прямо в
  этой системе (`apt install docker.io` в Debian/Ubuntu). Docker Desktop при
  этом не требуется.
- **Запуск Windows-бинарником из-под самой Windows** — удобнее всего Docker
  Desktop с включённой WSL-интеграцией: он поднимает демон и пробрасывает
  named pipe, к которому приложение подключается автоматически.

В любом случае Docker должен работать в режиме **Linux-контейнеров** — образы
лабов линуксовые. Метрики CPU/RAM читаются на всех платформах и Docker не
требуют.

Мелкие отличия под Windows: завершение по `Ctrl+C` работает (`SIGINT`),
а `SIGTERM` система не доставляет — на функциональность не влияет. Эмодзи в
логах корректно отображаются в Windows Terminal; в старом `cmd.exe` могут
выглядеть некорректно.

## Структура проекта

```
.
├── main.go              # всё приложение
├── go.mod / go.sum      # зависимости
├── configs/             # конфиги лабораторий (читаются в рантайме)
│   └── labs.yaml
└── templates/           # HTML-интерфейс (вшивается в бинарник при сборке)
    ├── index.html       # dashboard оператора
    └── admin.html       # панель администратора
```

Важно, где что находится в момент работы:

- Папка `templates/` встраивается в бинарник директивой `//go:embed` **на этапе
  сборки** — рядом с готовым бинарником её держать не нужно.
- Папка `configs/` читается **в рантайме** относительно рабочего каталога, из
  которого запущено приложение. Она должна лежать рядом при запуске.

## Быстрый старт

```bash
# 1. Забрать зависимости
go mod tidy

# 2. Собрать
go build -o cyberrange .

# 3. Запустить (из каталога, где рядом лежит configs/)
./cyberrange
```

После старта в консоли появятся адреса:

```
🌐  Dashboard:    http://localhost:8080
📊  Admin Panel:  http://localhost:8080/admin
❤️   Healthcheck: http://localhost:8080/healthz
```

Запуск для разработки без сборки бинарника:

```bash
go run .
```

Под Windows бинарник будет называться `cyberrange.exe`
(`go build -o cyberrange.exe .`), запуск — `.\cyberrange.exe`. Следите, чтобы
в рабочем каталоге рядом лежала папка `configs/`.

## Переменные окружения

| Переменная    | По умолчанию | Назначение                                                        |
|---------------|--------------|-------------------------------------------------------------------|
| `PORT`        | `8080`       | Порт веб-сервера.                                                  |
| `LAB_BIND_IP` | `127.0.0.1`  | IP, на котором публикуются порты лабов. `0.0.0.0` открывает их наружу — только осознанно. |

Пример:

```bash
PORT=9000 LAB_BIND_IP=127.0.0.1 ./cyberrange
```

## Конфигурация лабораторий

Лабы описываются в `configs/*.yaml` (или `*.yml`). Поддерживаются два формата:

- один лаб на файл;
- несколько лабов в одном файле, разделённых строкой `---` (multi-document YAML).

Несколько лабов подряд **без** `---` — это невалидный YAML (повторяющиеся
ключи), такой файл будет отклонён с ошибкой в логе.

### Поля

| Поле          | Тип             | Обязательное | Описание                                             |
|---------------|-----------------|:------------:|------------------------------------------------------|
| `name`        | строка          | да           | Уникальное имя лаба (оно же имя контейнера).          |
| `image`       | строка          | да           | Docker-образ.                                         |
| `description` | строка          | нет          | Короткое описание для интерфейса.                     |
| `category`    | строка          | нет          | Категория (например `security`, `web`).              |
| `difficulty`  | строка          | нет          | `beginner` / `intermediate` / `advanced` (для бейджа).|
| `tags`        | список строк    | нет          | Метки.                                                |
| `ports`       | список          | нет          | Проброс портов (см. ниже).                            |
| `env`         | словарь         | нет          | Переменные окружения контейнера.                      |
| `volumes`     | список строк    | нет          | Монтирования в формате `хост:контейнер[:ro]`.         |
| `healthcheck` | объект          | нет          | Docker-native проверка (см. ниже).                    |

Порт задаётся парой `container`/`host`. Значения берите в кавычки, чтобы YAML
не спотыкался о двоеточие:

```yaml
ports:
  - container: "80/tcp"
    host: "8082"
```

### Пример

```yaml
name: dvwa-lab
image: vulnerables/web-dvwa
description: Damn Vulnerable Web Application
category: security
difficulty: intermediate
tags: [security, web, vulnerable]
ports:
  - container: "80/tcp"
    host: "8082"
env: {}
volumes: []
```

### Healthcheck (необязательно)

Начиная с host-side проверки доступности, эта секция **не обязательна** —
дашборд считает стенд «здоровым», как только его порт начинает отвечать с
хоста. Секцию имеет смысл добавлять, только если хочется видеть родной статус
Docker в `docker ps`.

```yaml
healthcheck:
  test: "wget -q -O /dev/null http://127.0.0.1:3000/ || exit 1"
  interval: 10s      # как часто проверять
  timeout: 5s        # сколько ждать ответа
  retries: 5         # сколько провалов подряд до статуса unhealthy
  start_period: 60s  # окно на старт, в которое провалы не считаются
```

Команда `test` выполняется **внутри** контейнера, поэтому используйте только
то, что есть в образе (`curl`/`wget`/`node`/`bash` — проверяйте). Для Node-
приложений надёжна проверка через сам `node`, а не через `wget`, которого в
slim-образах может не быть.

## HTTP API

| Метод | Путь                     | Назначение                                             |
|-------|--------------------------|--------------------------------------------------------|
| GET   | `/`                      | Dashboard оператора.                                   |
| GET   | `/admin`                 | Панель администратора.                                 |
| GET   | `/healthz`               | Состояние самого приложения (200 / 503).               |
| GET   | `/api/status`            | Метрики системы и статус контейнеров.                  |
| GET   | `/api/labs`              | Список настроенных лабов.                              |
| POST  | `/api/lab/:name/start`   | Запустить лаб.                                          |
| POST  | `/api/lab/:name/stop`    | Остановить и удалить контейнер лаба.                    |
| GET   | `/api/admin/stats`       | Статистика запусков по лабам.                          |

### Пример: `GET /api/status`

```json
{
  "system": { "cpu": 12.4, "ram": 38.1, "ram_used_mb": 6100, "ram_total_mb": 16000 },
  "containers": [
    {
      "lab_name": "juice-shop-lab",
      "status": "running",
      "health": "healthy",
      "docker_health": "unhealthy",
      "reachable": true,
      "ip": "172.17.0.3",
      "port": "8083",
      "url": "http://localhost:8083"
    }
  ]
}
```

Поле `health` — итоговое состояние для дашборда (основано на доступности
порта с хоста). `docker_health` — «сырой» статус внутреннего Docker
HEALTHCHECK, оставлен для отладки: он может расходиться с `health`, если
проба внутри контейнера сломана, а приложение при этом работает.

### Пример: `docker inspect` для отладки healthcheck

Если внутренняя проба Docker показывает `unhealthy`, хотя стенд открывается:

```bash
docker inspect --format '{{json .State.Health}}' <container> | python3 -m json.tool
```

В `Log[].Output` будет причина (например `not found` — нужного бинарника нет
в образе).

## Как это работает

Приложение поднимает HTTP-сервер на Gin и подключается к демону Docker через
официальный SDK. Конфиги лабов загружаются при старте из `configs/`.

Фоновая горутина-монитор раз в 3 секунды параллельно собирает метрики системы
(gopsutil) и опрашивает контейнеры, кэшируя результат под мьютексом. Для
каждого запущенного лаба выполняется TCP-проверка его опубликованного порта с
хоста — так статус отражает реальную доступность стенда, а не только вердикт
внутренней пробы Docker.

Запуск лаба (`/start`) сериализуется per-lab мьютексом, тянет образ только при
его отсутствии локально, создаёт и стартует контейнер, после чего ждёт, пока
порт не начнёт отвечать. По `Ctrl+C` сервер и фоновые горутины завершаются
корректно через отменяемый `context`.

## Известные ограничения

- Статистика запусков хранится в памяти и обнуляется при перезапуске.
- Запуск лаба с тяжёлым образом держит HTTP-запрос до нескольких минут
  (образ тянется синхронно). Для продакшена стоит сделать асинхронный старт.
- Метрики RAM/CPU — общесистемные (по хосту), а не по каждому контейнеру.
- Аутентификация в интерфейсе не предусмотрена — это демо для локального и
  изолированного использования.
